# 该不该存进 memory —— 正例 / 反例(拿不准时冷读)

> **冷测试**:设想只读得到 repo、读不到这条的新 AI 做相关任务——**会照样做错 → 存;能从 代码+注释+全局 rule+模型常识 推对 → 不存**。两闸:① 会复发(根因稳定不变量→会;一次性→否,只发生一次的即便被纠正也不存)② 推不出来(冷测试)。
> **路由(二元)**:能进出错点代码注释 → 写注释;写不进又推不出的不变量 → memory(末位,40 条实测仅 3 过闸)。**用户纠正 = 触发、非充分**:「我不知道」存 vs「知道没执行好」不存。教训落进 durable 处即 supersede。

## ✅ 正例(技术隐藏坑 + 业务模型 —— 踩了/读了代码都还会做错)
1. 嵌入式设备内置 HTTP server 单线程,并发会卡死/重启 → 必须串行。(硬件隐藏行为)
2. Figma 导出 SVG 用 CSS 变量/`<style>` 填色 → 移动端渲染器不解析 → 整图空白 → 改内联 `fill`。(工具导出陷阱)
3. OAuth 成功 ≠ 账号可用 → 本产品强制补绑手机号才算登录完成。(业务规则)
4. 某设备命令是场景 A 用的,场景 B 误调触发异常(指示灯报错/进错模式)。(API 误用陷阱)
5. 推拉流主体是「边缘设备」非「客户端 App」,App 只是遥控器。(业务模型,搞反整条链路全错)
- 也够格:商业/合规硬边界(被告知或踩了才知道)、项目已有能力(防重造轮子)。

## ❌ 反例(看着想记、其实噪音)
1. "AuthAPI 拆成 AuthAPI+WechatManager" → 文件改动,git log 已有。
2. "iOS 用 Combine、Android 用 StateFlow" → 架构范式,读代码/已在 CLAUDE.md。
3. "网络请求加超时重试" → 模型已会的通用常识。
4. "今天 build 通过、某功能联调完成" → 在途状态,下次会话无用。
5. "current-state 上限 8 条" → 纯风格决策,搞错重调即可、不坏事。

## ⚖️ 边界 + 口诀
❌ "选了播放库 A"(配置看得到) ↔ ✅ "**某设备的流用库 B 会卡死、必须用 A**"(坑+原因)。
> 能 grep/git log 出、或"搞错重写一下"的 → 反例;踩了才知道、违反就搞坏东西的 → 正例。拿不准 → 不存。
