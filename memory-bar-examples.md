# 该不该存进 memory —— 5 正例 / 5 反例(拿不准时冷读)

> **何时读**:写 memory / current-state / dev-cases 认知前,**拿不准这条够不够格** → 读这个,把手上的跟下面例子比"像哪边"。
> **判别两闸(全过才存)**:① **会复发** —— 第一次踩到就预判它属不属于「会再遇到的那一类」(根因 = 稳定不变量 → 会;一次性 → 否,**只发生一次的即便被纠正也不存**);② **推不出来** —— grep/git/lockfile/读代码/模型常识 都拿不到才算(硬件/软件坑、强业务知识、商业约束、项目优势属此类)。**高沟通成本 / 用户纠正是触发信号、非充分条件**:分清「我不知道」[存] vs 「我知道但那次没执行好」[不存]。**再按适用面路由**:仅本项目 → 项目 memory;跨项目个人 → `local/CLAUDE.md`;团队受益 → `rules/`(git 同事可见)。

## ✅ 正例(照这个级别才存 —— 技术隐藏坑 + 项目业务模型 两类)

1. **某嵌入式设备的内置 HTTP server 单线程,并发请求会把它卡死 / 重启 → 必须串行访问。**(硬件隐藏行为 / 设备崩 / 谁都会想当然地并发)
2. **设计工具(如 Figma)导出的 SVG 用 CSS 变量 / `<style>` 填色 → 原生 / 移动端渲染器不解析 → 整图空白 → 改内联 `fill`。**(工具导出陷阱 / 图标全没 / "导出即用"是默认)
3. **第三方登录(OAuth)成功 ≠ 账号可用 → 本产品强制补绑手机号才算完成登录。**(业务规则 / 漏了建出坏账号 / 直觉以为 OAuth 成功就登录完成)
4. **某设备命令是给场景 A 用的,场景 B 误调会触发异常状态(如指示灯报错 / 进错模式)。**(API 误用陷阱 / 设备异常 / 名字看着就"该调")
5. **推拉流主体是「边缘设备」不是「客户端 App」**:App 只是遥控器,设备负责推 / 拉流;只有普通参与者才用 App 推流。(业务模型 / 搞反整条链路全做错 / 直觉以为像普通 App 客户端直接推流)

## ➕ 也够格(同标准,扩类)

- **商业环境限制**:如某平台审核政策禁止某做法 → 功能必须绕开(合规/政策硬边界,被告知或踩了才知道)。
- **项目优势 / 能力**:如"已有 X 能力/资源,别重复造"(防 AI 低估能力或重造轮子)。
- **高沟通成本 / 用户纠正 = 触发信号(非充分)**:它提示"停下判断一下",但仍要过两闸 —— 分清是「我不知道」(过②闸 → 存)还是「我知道、只是那次没执行好」(泛化指令,不存)。

## ❌ 反例(不存 —— 看着想记,其实是噪音)

1. "本会话把 AuthAPI 拆成 AuthAPI + WechatManager" → 文件改动,**git log 已有**。
2. "iOS 用 Combine、Android 用 StateFlow" → 架构范式,**读代码 / 已在 CLAUDE.md**。
3. "current-state 上限定为 8 条" → 纯风格决策,**搞错重调即可,不坏事**。
4. "网络请求要加超时和重试" → **模型训练已会的通用常识**。
5. "今天 build 通过、某功能联调完成" → **在途状态 / 做了什么**,下次会话无用。

## ⚖️ 边界对照(困惑就发生在这条线上)

- ❌ "选了播放库 A 不选 B"(纯决策,依赖配置里看得到) ↔ ✅ "**某设备的流用库 B 会卡死,必须用库 A**"(坑 + 原因)。
- ❌ "关了代码混淆"(配置看得到) ↔ ✅ "**开了混淆会让 API / 设备通信异常,release 必须关**"(违反就坏事的原因)。

> **同一件事:记"踩了才知道、违反就坏事"的原因,不记"看得到的决策/配置/做了什么"。**
> 口诀:能 grep / git log 出来、或"搞错重写一下"的 → 反例;踩了才知道、违反就搞坏东西的 → 正例。拿不准 → 不存。
